## Approach

Output-neutral WORK reduction targeting the newly-added Context Tree Weighting model (the largest single WORK contributor: it adds ~3.5G, doing ~4 HashMap lookups x 32 depths in predict and ~33 get+insert in update, every bit). The node store HashMap<u64, Node> used the default SipHash, but the key() function already applies a strong avalanche mix (multiply + xor-shift), so SipHash is redundant work on every lookup. Swapped the hasher to a BuildHasherDefault<IdHasher> that simply returns the u64 key. HashMap lookups still match by exact key, so every node read/written — and therefore every CTW prediction — is byte-for-byte identical. SCORE stays 572577 (the current record); WORK 12.49G -> 10.22G (-18.2%) on scripts/measure-complexity.sh; the adversarial round-trip suite passes. No algorithm/model/precision change whatsoever — only the hash function used to index an exact map.
Because SCORE is unchanged at the record value, this also lowers the leaderboard's complexity tiebreak (WORK) for that score.
