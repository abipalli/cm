## Approach

Memory optimization, provably identical output. The direct-counter probability tables (cp) were initialized to 2048 (P=0.5), which eagerly writes every page and forces the OS to commit the full ~1.3 GB even for slots a 384 KB file never touches. Storing cp as the offset (prob - 2048) in i16 lets the tables be zero-initialized, so allocation uses calloc and the OS commits only pages that are actually written. Reads add 2048, writes subtract it — values are byte-for-byte identical, so the predict/update sequence and SCORE are unchanged (578672). The other large tables (cn, st, match) were already zero-initialized; this brings cp in line.
Re an external on-disk index: not pursued — the rules require compress/decompress to use only their input (no file/side-channel reads, decompress reconstructs purely from the compressed bytes), and a precomputed index would also be disk-I/O-bound rather than faster. This is the in-RAM equivalent benefit.
